## F.1 General Terms

### API

**Application Programming Interface.** A software interface through which one application communicates with another.

### Backend

The software layer responsible for processing business logic and data.

### Container

An isolated runtime environment managed by Docker. See also [Container](#container-1) in the Docker terms.

### Daemon

A background service that runs continuously without direct user interaction.

### Deployment

The process of installing or updating software in the production environment.

### Environment

The complete collection of hardware, software and configuration that supports an application.

### Host

The physical or virtual Linux server on which the platform runs.

### Infrastructure

The hardware, operating system, networking and software required to support production services.

### Production

The live operational environment used by real users.

### Service

A software component that performs a specific operational responsibility.

## F.2 Blockchain Terms

### Block

A validated collection of blockchain transactions.

### Blockchain

A distributed ledger composed of sequentially validated blocks.

### Block Height

The position of a block within the blockchain.

### Confirmation

A block added after the block containing a transaction. Higher confirmation counts increase confidence that the transaction is permanent.

### Hash

A cryptographic digest that uniquely represents a set of data.

### Node

Software that participates in the blockchain network. The platform uses a fully synchronized local eCash node.

### RPC

**Remote Procedure Call.** The interface applications use to communicate with the blockchain node.

### Transaction

A blockchain operation recorded within a validated block.

### TXID

**Transaction identifier.** A unique identifier assigned to every blockchain transaction.

### ZMQ

**ZeroMQ.** A messaging interface the blockchain node uses to notify applications about blockchain events.

---

## F.3 Mining Terms

### ASIC

**Application-Specific Integrated Circuit.** Dedicated hardware designed for cryptocurrency mining.

### Block Template

Candidate block data generated by the blockchain node for miners.

### CKPool

The mining server responsible for Stratum communication, share validation and block submission.

### Difficulty

A numerical representation of the work required to solve a block.

### Hashrate

The computational capacity of a mining device or mining pool.

### Miner

A hardware device that participates in blockchain mining.

### Share

Proof of computational work submitted by a miner. Shares contribute to mining statistics and may eventually produce a valid block.

### Solo Mining

Mining performed independently, without reward sharing among multiple participants.

### Stratum

The communication protocol between mining hardware and the mining server.

### Worker

An individual mining device registered under a miner account.

## F.4 Docker Terms

### Bind Mount

A mapping between a host directory and a container directory.

### Container

A lightweight, isolated runtime environment. In this documentation, containers are managed by Docker.

### Docker Compose

A configuration format that defines containers, networks and volumes.

### Docker Engine

The runtime responsible for managing containers.

### Docker Network

An isolated communication network that connects containers.

### Docker Volume

Persistent storage managed directly by Docker.

### Image

A read-only template used to create containers.

### Restart Policy

The rule that controls automatic container restart after a failure.

---

## F.5 Networking Terms

### DNS

**Domain Name System.** Translates domain names into IP addresses.

### Firewall

A security component that controls permitted network traffic.

### Gateway

A network device or route that connects different networks.

### HTTP

**Hypertext Transfer Protocol.**

### HTTPS

Encrypted HTTP communication.

### Port

A logical communication endpoint used by network services.

### Reverse Proxy

A server that receives client requests and forwards them to backend services. The platform uses Caddy as its reverse proxy.

### TLS

**Transport Layer Security.** Provides encrypted network communication.

---

## F.6 Security Terms

### Authentication

Verification of a user or service identity.

### Authorization

Determination of the permissions granted to an authenticated entity.

### Certificate

A digital document that supports encrypted communication.

### Least Privilege

The security principle of granting only the permissions required to perform a task.

### Private Key

A confidential cryptographic key used during secure communication.

### Secret

Confidential configuration information, such as passwords, API keys or authentication tokens.

---

## F.7 Blockchain Notary Terms

### Certificate PDF

A generated document that confirms blockchain notarization.

### Document Hash

The SHA-256 digest calculated from the original document. Only the hash is stored on the blockchain.

### Notarization

The process of proving that a document existed in a specific state at a specific time by anchoring its hash to the blockchain.

### SHA-256

The cryptographic hash algorithm used by the Blockchain Notary.

### SQLite

The embedded relational database used by the Blockchain Notary application.

### User Key

An application-level authentication value used by registered users.

---

## F.8 Operational Terms

### Backup

A protected copy of operational information used for recovery.

### Disaster Recovery

The process of restoring production services following a major failure.

### Health Check

A verification procedure that confirms production services are operating correctly.

### Maintenance Window

A scheduled period during which operational changes may be performed.

### Monitoring

Continuous observation of infrastructure and application status.

### Recovery

The process of restoring a failed component to normal operation.

### Rollback

The process of returning the platform to a previously known stable configuration.

### Runbook

A documented operational procedure executed by administrators.

## Release 1.0 Editorial Notes

Release 1.0 consolidates the Appendix F RC2 source into the definitive glossary and acronym reference. The editorial review:

- standardized release metadata, heading hierarchy, capitalization and punctuation;
- converted the table of contents to linked navigation;
- preserved every RC2 glossary entry and its documented technical distinctions;
- clarified concise definitions without adding production behavior or configuration;
- distinguished the general and Docker-specific uses of **Container**; and
- relabeled the acronym table to acknowledge entries that are product or protocol terms rather than initialisms.

### Unresolved Factual or Terminology Verification Items

The available documentation does not provide enough verified evidence to resolve the following terminology details without external confirmation:

- whether **CKPool** should be expanded or treated only as the software name;
- whether **Stratum** should remain in the acronym table despite being a protocol name rather than an acronym;
- whether the project intends **SQLite** to be described by a formal expansion rather than as the embedded database product; and
- whether **Certificate** in the security glossary is intended to mean specifically a TLS digital certificate.

These items remain editorial verification questions and do not change the documented production architecture.
